Folder-as-agent

An agent is a folder: a markdown role, a context file, some examples, a list of tools. Duplicate the folder, edit the markdown, you've made a new agent. No SDK, no framework, just filesystem conventions.

The bet. Agent definitions should be as easy to fork as a Google Doc. The moment agents require SDK scaffolding, you've limited who in a company can build one.

The argument

Every is the defining practitioner of this school — their operational writing about their own company is the primary source material. The title essay, The Folder Is the Agent (Every, 2026-04-13), describes a working setup of 44 agents, each a folder in the repo. The pattern is deliberately zero-API: role, context, examples, and allowed tools live in plain markdown so anyone on the team can create or fork an agent without engineering support.

The companion writing builds out how the pattern actually runs day-to-day. House Rules for the Agents (Every, 2026-04-05) documents the organization-wide conventions that every folder inherits. Every Is Half Agent Now (2026-04-08) is the etiquette piece — what it feels like when half your colleagues are markdown folders, and where the human adds value anyway. How We Run a 25-Person Company on Four AI Agents (2026-04-09) connects folder-as-agent to org design, showing how the pattern scales organizationally.

Folder-as-agent overlaps substantively with context engineering (the folder is the curated context) and with compound engineering (folders are the stackable units). What distinguishes it is the anti-framework stance: the primitive is a directory, not a class, not a graph, not a server.
