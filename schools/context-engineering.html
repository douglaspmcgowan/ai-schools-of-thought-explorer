Context engineering

The architecture is the prompt. The real leverage is in what you put in the context window: which files, which memories, which examples, which tool outputs. Framework choice matters less than curation quality.

Sits next to "single-loop". Single-loop says: keep the control flow simple. Context engineering says: and spend the saved effort on what goes into that loop.

The argument

The canonical move of this school is treating the filesystem as the agent's working memory. "The File System Is the New Database" names the pattern: a folder of markdown becomes the long-term store because the agent can read it on demand. Tools like Grov generalize this into shared reason memory across Claude Code sessions — a concrete expression of the thesis that the substrate around the model is where the engineering belongs.

The reason this matters empirically is that in-context learning is fragile. Tencent's research on "Learning from context is harder than we thought" documents how small changes in ordering, phrasing, and example selection produce large behavior swings. That fragility is why curation is the craft. The "Expensively Quadratic" essay makes the cost argument: context grows, token spend grows faster, so the quality-per-token of what you include is the actual optimization target.

Adjacent work extends the frame: the Codex Prompting Guide and Addy Osmani's "Agentic Engineering" both lean heavily on context curation as the practitioner's main lever.
