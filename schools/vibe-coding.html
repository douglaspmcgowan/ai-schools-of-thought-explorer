Vibe coding

Karpathy's term. Describe the vibe, let the model write it, don't read the diff. Not for production — for sketching, prototyping, personal tools. The point is speed of exploration, not correctness.

Often misread. Vibe coding isn't a claim that code review is obsolete. It's a claim that there are many software-shaped problems — internal tools, one-off analyses, personal utilities — where the cost of careful coding used to exceed the value. Vibe coding unlocks those.

The argument

The strongest empirical corrective is SaaStr's "What Folks Are Really Vibe Coding Today", which surveys what people actually build: internal dashboards, demo apps, quick automations. Not replacements for Salesforce. The vibe-coding practice is niche but high-leverage — it expands the population of people who can produce working software for problems that didn't previously justify engineer time.

The intellectual companion piece is "Sketching with Code" (Proof of Concept), which reframes vibe coding as a design practice — code as charcoal, cheap and discarded. That framing is useful for anyone trying to justify the practice to skeptics: vibe coding is not a replacement for engineering, it's an expansion of the creative-exploration toolkit. App-builder tools (Google Opal, v0, Bolt, Lovable) are the productized form of the same move.
